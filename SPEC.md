## 1. Scope and conformance

RepoPact is a repository-native standard for durable agent work: authority,
intent, evidence, and drift are recorded as versioned files so project state is
recoverable without a prior conversation.

An implementation is **RepoPact-conformant** if, given a repository tree, it
accepts exactly the repositories that satisfy every rule in §3–§7 and rejects the
rest. `scripts/validate_repo.py` is the **reference implementation**; where this
prose and the reference implementation disagree, that is a defect in one of them
and is resolved by an audit finding, not by silent divergence.

## 2. Repository layout

```
AGENTS.md                      root contract (required)
VERSION                        semantic project version (required)
governance/charter.md          principles and invariants (prose)
governance/invariants.json     binding invariants (required)
governance/frozen-surface.json protected paths and symbols (required)
governance/owners.json         scopes, roles, concurrency (required)
governance/policies/           durable operating rules
schemas/                       record schemas (required)
decisions/                     decision records (ADRs)
work/{active,blocked,deferred,completed}/  work items
evidence/runs/                 evidence manifests
audits/registry.json           audit coverage and cadence (required)
audits/findings/               audit findings
audits/reports/dashboard.md    derived (generated)
```

## 4. Validation rules (semantics)

These are the rules JSON Schema cannot express; they are authoritative for
cross-record consistency.

1. **Contract coverage.** The root `AGENTS.md` exists. Every nested `AGENTS.md` is
   registered in `audits/registry.json`. An `_audit/` companion is optional, but if
   present must contain `README.md`, `inventory.md`, and `alignment-report.md`.
2. **Identity agreement.** A record's `id`, its directory or filename prefix, and
   (for work items) its lifecycle directory must agree. IDs are unique per type.
3. **Reference resolution.** Work-item `depends_on` and acceptance-criterion
   `evidence`, evidence `work_item`, finding `scope`, role `scopes`, and decision
   `supersedes` must reference records or scopes that exist.
4. **Acceptance.** A `satisfied` criterion has linked evidence. A `completed` work
   item has no `pending` criterion.
5. **Acyclic dependencies.** The work-item `depends_on` graph has no cycles.
6. **Concurrency (optional).** When `owners.json` enables
   `enforce_disjoint_active_scopes`, no two non-terminal work items share a scope.
7. **Version.** `VERSION` is `MAJOR.MINOR.PATCH`.

## 5. Lifecycle state machine

A work item's status is the name of its lifecycle directory; the JSON `status` must
match. States: `active`, `blocked`, `deferred`, `completed`. A transition moves the
entire directory and updates `status`; the reasoning, decisions, and evidence links
are never dropped. `completed` requires every acceptance criterion `satisfied` or
`waived` with evidence. Closed work is never rewritten to look cleaner (INV-4).

## 6. Invariants and escalation

Invariants are binding: an agent may not weaken one without flagging it and
obtaining explicit human operator approval. Each declares a `rationale`, an
`escalation` path, and an `enforced_by` (a machine rule, or `null` when only human
review can enforce it).

<!-- generated:invariants -->
| ID | Statement | Enforced by |
| --- | --- | --- |
| INV-1 | No critical state exists only in conversation; authority, intent, decisions, evidence, and history live in versioned files. | human review (escalation) |
| INV-2 | A completed work item has zero pending acceptance criteria. | scripts/validate_repo.py:validate_work (completed item has pending criterion) |
| INV-3 | A satisfied acceptance criterion has linked evidence. | scripts/validate_repo.py:validate_work (satisfied without evidence) |
| INV-4 | Completed work is never rewritten to make history appear cleaner. | human review (escalation) |
| INV-5 | The deepest applicable AGENTS.md refines its parents but cannot weaken them. | human review (escalation) |
| INV-6 | Changes to the declared frozen surface require explicit human operator approval. | scripts/check_frozen_surface.py |
| INV-7 | Derived artifacts are generated from source records, never hand-edited as if they were sources. | .github/workflows/governance.yml (dashboard diff check) |

## 7. Frozen surface

`governance/frozen-surface.json` lists protected `glob` paths and `symbols`. A
change touching a protected path or symbol requires operator approval (INV-6).
`scripts/check_frozen_surface.py` is the diff-time backstop; it is best-effort and
reports rather than silently blocking, because the binding gate is human review.

## 8. Versioning

`VERSION` is the semantic project version. Release tags may carry a maturity label
(e.g. `v0.1.0-alpha`). Backward-incompatible changes to any record schema or to a
§4 rule require a MAJOR bump and a decision record.

## 9. Conformance and reference implementation

Run the reference implementation:

```
pip install -r requirements.txt
python scripts/validate_repo.py
```

A conformant alternative implementation must produce an equivalent accept/reject
decision on the same repository. RepoPact governs its own development, so this
repository is the canonical conformance example.

A machine-checkable corpus lives in [`tests/fixtures/`](tests/fixtures/): a valid
baseline that must be accepted, and one invalid overlay per rule above that must be
rejected with a declared message. `tests/test_conformance.py` exercises it; an
alternative implementation can target the same fixtures.
